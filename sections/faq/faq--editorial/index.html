Everything here is a single copy-paste block — no build step, no dependencies beyond one Google Fonts link. Here's what people ask before dropping a section in.

Still stuck on something?
Write to hello@sectionlibrary.dev — a human reads it.

01 Do I need Tailwind, React, or any build tooling to use these?

No. Every section is one self-contained index.html file: a <style> block, the markup, and nothing else. Paste it into Claude Code, open it in a browser, and it renders.

The only external request is the Google Fonts link in the head. Cut that line and swap in a system font stack if you want a truly zero-network file.

02 Can I change the colours and fonts without breaking anything?

Yes — that's the point of the :root{} token block at the top of every file. Edit --accent, --ink, or --bg once and the whole section re-tunes itself.

Fonts are two tokens too: --serif and --sans. Change the Google Fonts URL and the variables, and you never touch a single component rule.

03 Are the sections responsive on phones and tablets?

Every file ships with a @media(max-width:640px) block that collapses multi-column layouts, resizes headings, and reflows spacing. Designed at ~1440px, checked down to 320px.

04 What's the licence — can I use this in client work?

Free to use, modify, and ship — including paid client projects. No attribution required, though a mention is always kind.

The one line you can't cross: don't repackage the library itself and resell it as a competing template pack.

05 How do I combine several sections into one full page?

Keep one file's <head> and :root{} tokens as your base, then paste the <section> markup from each piece in order. Since tokens are shared, colours and type stay consistent across the whole page automatically.

If two sections define the same class name, prefix one — a faq__ style convention keeps collisions rare by design.

06 Do you take requests for new section types?

Often, yes. Pricing tables, testimonial walls, and comparison blocks all started as reader requests. Send the use case and a rough sketch — the clearer the brief, the faster it lands in the library.
