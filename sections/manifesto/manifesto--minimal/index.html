How we build — no exceptions

Speed isn't a feature you add later. It's a decision you make on day forty day one.

We've shipped enough software to know where the rot starts. So we wrote it down — five things we refuse to negotiate, even when the deadline says otherwise.

01
Ship the boring version first.
The clever abstraction can wait. Working code that a teammate can read at 2am is worth more than an elegant one nobody dares touch.
02
A slow page is a broken page.
We don't file performance under "polish." If it makes someone wait, it isn't finished — no matter how good it looks.
03
Defaults are a promise.
Whatever the tool does out of the box is what most people will live with forever. So we sweat the default more than the setting.
04
Delete before you add.
Every feature is a tax someone pays later. The best pull request of the quarter usually has a red diff, not a green one.
05
Say the hard thing early.
Bad news doesn't age well. We'd rather lose an argument in week one than lose a launch in week nine.
